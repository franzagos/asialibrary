# Codex Compatibility Checklist

A maintainer-facing smoke test. Walk through this in a real Codex CLI session to verify the repo's claims about Codex support. Each section is a pass/fail item you can record.

**Setup:** fresh clone of the boilerplate, in a Codex-configured terminal, with a working shell (`pnpm --version`, `codex --version`).

---

## 1. Project rules are discovered

Start a Codex session inside the repo root.

- [ ] Ask Codex: *"What is this project, and what conventions must I follow?"*
- [ ] **Expected**: response references content from `AGENTS.md` (tech stack, rules about `pnpm lint`/`pnpm typecheck`, API utils, etc.). The assistant should know this without being told to load a file.
- [ ] Record: which file did the agent actually load? `AGENTS.md`, something else, or nothing?

## 2. Shared commands are reachable (either via discovery or direct feeding)

### 2a. Try discovery

- [ ] Type `/starter-prompt` or `/create-spec` as a Codex slash command.
- [ ] **Pass**: Codex recognizes the command and starts executing the `.shared/commands/{name}.md` instructions.
- [ ] **Fail**: Codex returns "unknown command" — proceed to 2b.

### 2b. Direct feeding (fallback)

- [ ] Open `.shared/commands/create-spec.md` and paste the contents into Codex, with a prefix like: *"Please act on these instructions as a custom slash command would. Here's the content:"*
- [ ] **Expected**: Codex follows the command instructions the same way it would a native slash command.

Either 2a or 2b must work. If neither does, the shared command layer is not viable in your Codex setup — file an issue.

## 3. Sequential mode works end-to-end (no sub-agents)

This is the critical one.

- [ ] Run `/create-spec` (or feed its content) for a *tiny* test feature. Ask for a spec about, e.g., "add a read-only Settings page that shows the signed-in user's email."
- [ ] Decline the docs research if asked (or note that no docs tool is attached).
- [ ] **Expected**: Codex produces
  - `specs/settings-page/requirements.md`
  - `specs/settings-page/implementation-plan.md`
  - `specs/settings-page/action-required.md`
  - `specs/settings-page/decisions.md`
  - `specs/settings-page/research-notes.md` with a clear "Research Status" line
- [ ] **Expected**: the `Research Status` line in `research-notes.md` says `Execution mode: sequential` (or similar) — because Codex is not running parallel sub-agents in this smoke test.

Then:

- [ ] Run `/continue-feature` (or feed its content).
- [ ] **Expected**: Codex prints `Phase 1: ... · Wave 1 (N task(s), sequential)` and executes each task inline.
- [ ] **Expected**: at the end, `docs/features/settings-page.md` exists.
- [ ] **Expected**: all checkboxes in `specs/settings-page/implementation-plan.md` are `- [x]`.

## 4. Skills are recognized

- [ ] Ask Codex: *"Can you run a security review on this project?"*
- [ ] **Expected**: Codex loads the `security-audit` skill content from `.agents/skills/security-audit/SKILL.md` (or equivalent for your runtime) and produces a confidence-based report.
- [ ] **Expected**: no fixes are applied until you explicitly confirm.

## 5. MCP docs tool is optional

### 5a. With Context7 (or similar) attached

- [ ] Start `/create-spec` for a feature that needs an external API (e.g. "add Stripe checkout").
- [ ] **Expected**: `research-notes.md` has a filled-out External API Notes section with auth method, key endpoints, gotchas.
- [ ] **Expected**: `Research Status` shows `Docs Fetcher: complete`.

### 5b. Without any MCP docs tool

- [ ] Start `/create-spec` for the same feature.
- [ ] **Expected**: the command continues without complaining about missing tools.
- [ ] **Expected**: `research-notes.md` External API Notes section says something like "External API docs: skipped (no docs tool available). Verify vendor docs before implementing Stripe."
- [ ] **Expected**: `Research Status` shows `Docs Fetcher: skipped (no docs tool)`.

## 6. Security audit does not auto-fix

- [ ] Run `/security-audit` (or feed the command file).
- [ ] **Expected**: Codex produces the full report and asks: "Want me to fix the Critical and High items?"
- [ ] **Expected**: no files change until you answer "yes".
- [ ] If you answer "yes": Codex applies fixes in severity order and runs `pnpm lint && pnpm typecheck` after each batch.

## 7. Symlink structure

- [ ] `ls -la .agents/commands` resolves to `../.shared/commands`
- [ ] `ls -la .agents/skills` resolves to `../.shared/skills`
- [ ] `ls -la CLAUDE.md` resolves to `AGENTS.md`
- [ ] `ls -la .cursor/rules/project.mdc` is a regular file (generated by `scripts/sync-cursor-rules.sh`)

## 8. `pnpm check` passes on a clean clone

- [ ] `pnpm install`
- [ ] `pnpm check`
- [ ] **Expected**: lint + typecheck pass with zero errors.
